# Технические требования и примечания
1. В текущей версии API все запросы делаются методом GET, ответ приходит в виде JSON объекта. 
> Да, мы прекрасно знаем что это нехорошо а также что такое REST, с чем его едят и постоянно используем данную методологию для наших новых решений. API для партнёров сложилась в определённый исторический момент, обросла фичами и будет кардинально переработана как на предмет формата общения с API (тот самый вдумчивый RESTful), так и с точки зрения алгоритмов. 
2. В ответе всегда присутствует 2 параметра:
- status (базово имеет значение 'error' или 'success'. Однако в некоторых запросах может быть расширен до большего количества вариантов значений).
- status_text (текстовое описание результата, позволяет во время ручного дебага понять суть ответа).
3. Каждый GET параметр, передаваемый в запросе необходимо преобразовывать функцией urlencode()
4. Для удобства отладки, вы можете при запросе использовать дополнительный параметр array=1, при указании которого ответ будет отображен не в виде JSON а в виде человекопонятного массива.
5. Изначально для партнёра не активированы все направления. По всем вопросам подключения направлений обращайтесь к менеджеру по направлениям [@georgie_white](https://t.me/georgie_white)
6. Все направления, курсы и лимиты, с которыми мы работаем вы можете посмотреть по [ссылке](https://mine.exchange/request-exportxml.xml), однако для получения курсов использовать этот URL некорректно, используйте запрос "[rates](#запрос-rates-получение-текущих-курсов-ограничений-минимальноймаксимальной-суммы-и-резервов)".

## Контроль сетевых потерь
Связь между серверами хоть и редко, но может обрываться, поэтому необходимо валидировать, являются ли полученные в ответ данные ожидаемыми вами:
1. API всегда отвечает JSON ответом. Если вы не получили корректный JSON — вы не можете точно знать, прошёл ли ваш запрос или был потерян по пути. В случае критичных запросов (например, выплата), следует перепроверять такие запросы любым доступным вам способом и не считать запрос выполненным или не выполненным, пока не получите точного подтверждения произошедшего. 
- В некоторых случаях вы можете отправить уточняющий API запрос для проверки состояния. Например, если вы делали запрос на выплату и не знаете, прошёл ли он, вы можете проверить состояние методом "[masscheck](#запрос-masscheck-проверка-состояния-заказов)".
- В случае невозможности автоматизированной проверки, следует обратиться в техподдержку. Наша поддержка действительно оперативна, поэтому при возникновении такой исключительной ситуации, вы получите ответ максимально быстро.
2. После того как вы убедились что получили валидный JSON ответ, следует проверить параметр "status". Если status = error, вы можете быть уверены что запрос не был выполнен.

## Типичные сценарии работы:

### Типичное прохождение по одному заказу выглядит следующим образом:
1. [Запрос "create" (создание заказа)](#запрос-create-создание-заказа)
2. [Запрос "masspay" (подтверждение оплат с внутреннего счёта)](#запрос-masspay-подтверждение-оплат-с-внутреннего-счёта) — только для выплат с внутреннего счёта. Рекомендуется делать подтверждение выплат по планировщику (например, cron) и подтверждать сразу множество скопившихся заказов, вместо подтверждения по одному заказу.
3. [Запрос "masscheck" (проверка состояния заказов)](#запрос-masscheck-проверка-состояния-заказов) — рекомендуется делать проверку статусов по планировщику (например, cron) и проверять сразу множество скопившихся заказов, вместо проверки по одному заказу.

# Запрос "create" (создание заказа)

Данный запрос создаёт заказ, но не реализует непосредственно оплату. После создания заказа, для проведения обмена необходимо совершить оплату по реквизитам, выданным в ответе. Для внутреннего счёта (при хранении средств партнёра на нашем сервисе), оплата производится запросом ["masspay" (подтверждение оплат с внутреннего счёта)](#запрос-masspay-подтверждение-оплат-с-внутреннего-счёта).

## Параметры запроса
**Endpoint:** https://api.mine.exchange/create/

| **Параметр** | **Тип** | **Значение или описание** |  
|--|--|--|
| **key** | string | Ваш ключ авторизации API. |
| **currency_send** | string | [Код валюты](https://mine.exchange/request-exportxml.xml), обмен которой вы производите. |
| **currency_receive** | string | [Код валюты](https://mine.exchange/request-exportxml.xml), обмен на которую вы желаете совершить. |
| **account_send** | string | Счёт, с которого вы делаете оплату. Для некоторых валютных пар данный параметр не является обязательным. |
| **account_receive** | string | Счёт, на который вы получите средства. |
| **amount_send** или **amount_receive** | numeric | Сумма, которую вы оплачиваете или сумма, которую вы желаете получить. Необходим один из двух параметров. |
| **extra** | Object | На данный момент этот параметр является обязательным для приёма QIWI. |

### Дополнительные параметры для приёма QIWI
Платёжная система QIWI требует некоторые аналитические данные для борьбы с несанкционированными платежами. Для приёма QIWI требуется дополнительно указать следующие параметры:
| **Параметр** | **Значение или описание** |  
|--|--|
| **key** | Ваш ключ авторизации API. |
| <sup><sub>extra.</sub></sup>**client-ip** | IP вашего пользователя, который совершает действие. |
| <sup><sub>extra.</sub></sup>**client-user-agent** | Useragent (браузер) пользователя.<br><br> Примечание для php разработчика: содержится в `$_SERVER['HTTP_USER_AGENT']`. |
| <sup><sub>extra.</sub></sup>**client-email** | Email адрес пользователя. |
| <sup><sub>extra.</sub></sup>**client-id** | ID пользователя в вашей базе данных. |
| <sup><sub>extra.</sub></sup>**order-id** | ID заказа в вашей базе данных. |

Впоследствии hash используется при [запросе "masspay"](#запрос-masspay-подтверждение-оплат-с-внутреннего-счёта)..  |

##  Каждый элемент объекта "payment_variants":
Каждый элемент объекта содержит следующие параметры:
- error (Обязательный параметр. По умолчанию null, нет ошибки. В противном случае текстовое описание ошибки, из за которой невозможно использование данного метода оплаты).
- Один из следующих реквизитов оплаты, обязательный параметр:
    - url: адрес платёжной системы, куда необходимо переадресовать клиента. 
    - number: реквизиты, по которым необходимо перевести оплату.
    - id: используется в объекте hash как уникальный хеш транзакции для последующего подтверждения оплаты.
- note (Необязательный параметр. Текстовое примечание для упрощения интеграции программистам и упрощения процесса интеграции.)

### Реквизиты payment_variants в зависимости от приёмной платёжной системы
| **Платёжная система** | **Объект в payment_variants** | **Описание** |
|--|--|--|
| **QIWI** (QWRUB)                          | invoice | (url) Ссылка для переадресации клиента на выставленный счёт |
| **QIWI** (QWRUB)                          | direct  | (url) Ссылка для переадресации клиента на страницу прямой оплаты на кошелёк |
| **BITCOIN** (BTC)                         | manual  | (number) Номер btc кошелька для оплаты |
| **Внутренний счёт** (VNRUB, VNUAH, VNBTC) | hash    | (id) Уникальный хеш транзакции для последующего подтверждения оплаты с внутреннего счёта (когда клиент хранит средства в нашей системе). Впоследствии hash используется при [запросе "masspay"](#запрос-masspay-подтверждение-оплат-с-внутреннего-счёта). |
| **Остальные мерчанты**                    | direct  | (url) Ссылка на платёжную форму мерчанта |

**Внимание: Для транзакций источником которых не является внутренний счёт, данный метод использовать НЕ НУЖНО.**

**Endpoint:** https://api.mine.exchange/masspay/

| **ids** | string | Номера заявок через запятую (order1_id, order2_id...) |
| **hashes** | string | Хеши заявок (hash1, hash2...) |

| **order_error**<br>*Неизвестный* | Ошибка в заказе. Свяжитесь с технической поддержкой. **Повторную отправку делать запрещено.** |
| **payout_error**<br>*Неизвестный* | Ошибка выплаты, пожалуйста свяжитесь с службой поддержки. Данный статус **НЕ является финальным**. Средства возможно ушли получателю! Для уточнения деталей следует обратиться в поддержку нашего сервиса! |
| **delete**<br>*Финальный* | Заказ удален. |
| **cancel**<br>*Финальный* | Пользователь отменил заказ. В направлениях обмена c внутреннего счета данный статус означает, что заказ получил финальный статус, деньги по заказу не отправлены и возвращены на ваш внутренний счет. Возможные причины: ошибка в реквизитах получателя, ограничения банка эмитента и иные причины. После получения этого статуса можно повторять отправку. |
| **verify** | Заказ проверяется оператором сервиса. Свяжитесь с службой поддержки чтобы уточнить детали. |
| **pay_promised**<br>*Информационный статус, не требует реагирования* | Клиент отметил заказ как оплаченный, ожидается подтверждение от системы. |
| **coldpay**<br>*Информационный статус, не требует реагирования* | Ожидание проверки оплаты клиентом от системы. |
| **realpay** | Подтверждение оплаты клиентом получено. |
| **payout_process** | Ожидание выплаты средств. |
